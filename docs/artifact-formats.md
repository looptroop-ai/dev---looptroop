# Artifact File Formats

LoopTroop uses file-based artifacts stored in the `.ticket/` directory within each ticket worktree. These are the authoritative formats.

## interview.yaml

Generated by the council interview phase. Contains questions and user answers.

**Location:** `<worktree>/.ticket/interview.yaml`

**Schema:**
```yaml
schema_version: 1
ticket_id: "<ticket-id>"
artifact: "interview"
status: "approved"              # draft | approved
generated_by:
  winner_model: "<provider/model>"
  generated_at: "<ISO-8601 timestamp>"
questions:
  - id: "<Q1, Q2, ...>"
    prompt: "<question text>"
    answer_type: "<free_text | single_choice | multi_choice | boolean>"
    options: []                 # populated for single_choice and multi_choice types
    answer:
      skipped: false            # true if user skipped this question
      selected_option_ids: []   # populated for choice-type answers
      free_text: "<user answer or empty string>"
      answered_by: "<user | ai_skip>"
      answered_at: "<ISO-8601 timestamp>"
follow_up_rounds: []            # additional question rounds from coverage gaps
summary:
  goals: []
  constraints: []
  non_goals: []
approval:
  approved_by: "user"
  approved_at: "<ISO-8601 timestamp>"
```

## prd.yaml

Product Requirements Document generated from the interview results.

**Location:** `<worktree>/.ticket/prd.yaml`

**Schema:**
```yaml
schema_version: 1
ticket_id: "<ticket-id>"
artifact: "prd"
status: "approved"              # draft | approved
source_interview:
  content_sha256: "<sha256 of interview.yaml>"
product:
  problem_statement: "<...>"
  target_users: []
scope:
  in_scope: []
  out_of_scope: []
technical_requirements:
  architecture_constraints: []
  data_model: []
  api_contracts: []
  security_constraints: []
  performance_constraints: []
  reliability_constraints: []
  error_handling_rules: []
  tooling_assumptions: []
epics:
  - id: "EPIC-1"
    title: "<epic title>"
    objective: "<...>"
    implementation_steps: []
    user_stories:
      - id: "US-1"
        title: "<story title>"
        acceptance_criteria: []
        implementation_steps: []
        verification:
          required_commands: []
risks: []
approval:
  approved_by: "user"
  approved_at: "<ISO-8601 timestamp>"
```

## codebase-map.yaml

Auto-generated project structure map. Created by SYS when a ticket transitions from DRAFT to COUNCIL_DELIBERATING.

**Location:** `<worktree>/.ticket/codebase-map.yaml`

**Schema:**
```yaml
ticket_id: "PROJ-1"
artifact: "codebase_map"
manifests:
  - "package.json"
  - "pyproject.toml"
files:
  - "src/auth/LoginForm.tsx"
  - "src/auth/useAuth.ts"
```

**Generation rules:**
- Paths-only map (no directory entries)
- Stack-agnostic manifest discovery
- Max scan depth: 4 levels
- Ignores: `.git`, `node_modules`, `dist`, `build`, `.looptroop`, etc.

## issues.jsonl

Implementation beads (task graph). One JSON object per line. This is the **authoritative bead tracker** — not SQLite.

**Location:** `<worktree>/.ticket/beads/<flow-id>/.beads/issues.jsonl`

**Bead fields (22 per object):**

| # | Field | Type | Description |
|---|-------|------|-------------|
| 1 | `id` | string | Hierarchical bead ID with 4-char suffix hash (e.g. `PROJ-1-EPIC-1-US-1-task1-h7qd`) |
| 2 | `title` | string | Short task name |
| 3 | `priority` | integer | Execution order (sequential) |
| 4 | `status` | string | `pending` → `in_progress` → `done` / `error` |
| 5 | `issue_type` | string | `task`, `bug`, `chore`, etc. |
| 6 | `external_ref` | string | Parent ticket ID |
| 7 | `prd_references` | string | Epic/User Story mapping summary |
| 8 | `labels` | string[] | Must include at least one user story and epic label |
| 9 | `description` | string | Detailed implementation steps |
| 10 | `context_guidance` | object | `{ patterns: [...], anti_patterns: [...] }` |
| 11 | `acceptance_criteria` | string | Definition of done |
| 12 | `dependencies` | object | `{ blocked_by: [...], blocks: [...] }` |
| 13 | `target_files` | string[] | Files targeted by this bead |
| 14 | `tests` | string[] | Bead-scoped test descriptions |
| 15 | `test_commands` | string[] | Exact commands to run bead tests |
| 16 | `notes` | string | Append-only errors/learnings from retries |
| 17 | `iteration` | integer | Starts at 1, incremented on retry |
| 18 | `created_at` | ISO 8601 | Bead creation timestamp |
| 19 | `updated_at` | ISO 8601 | Last update timestamp |
| 20 | `completed_at` | ISO 8601 | When status set to `done` |
| 21 | `started_at` | ISO 8601 | When status set to `in_progress` |
| 22 | `bead_start_commit` | string | Git SHA at bead execution start |

**Example line:**
```json
{"id":"PROJ-1-EPIC-1-US-1-task1-h7qd","priority":1,"title":"Implement login error state","status":"pending","issue_type":"task","external_ref":"PROJ-1","prd_references":"EPIC-1 / US-1","labels":["ticket:PROJ-1","epic:EPIC-1","story:US-1"],"description":"Add inline error handling...","context_guidance":{"patterns":["Use AppError class"],"anti_patterns":["Do not use alert()"]},"acceptance_criteria":"Show non-blocking inline message","dependencies":{"blocked_by":[],"blocks":["PROJ-1-EPIC-1-US-1-task2-z8p1"]},"target_files":["src/auth/LoginForm.tsx"],"tests":["Login error banner appears"],"test_commands":["npm test -- --grep \"login error\""],"notes":"","iteration":1,"created_at":"2026-02-06T16:10:00Z","updated_at":"2026-02-06T16:10:00Z","completed_at":"","started_at":"","bead_start_commit":""}
```

## execution-log.jsonl

Append-only operational log for debugging and audit.

**Location:** `<worktree>/.ticket/runtime/execution-log.jsonl`

**Event types:** `state_change`, `model_output`, `test_result`, `error`, `bead_complete`

**Note:** This is non-authoritative — it's for debugging only, not for state recovery.

## Bead Completion Marker

When a bead finishes execution, the AI agent outputs a structured marker:

```
<BEAD_STATUS>{"bead_id":"PROJ-1-EPIC-1-US-1-task1-h7qd","status":"completed","checks":{"tests":"pass","lint":"pass","typecheck":"pass","qualitative":"pass"}}</BEAD_STATUS>
```

All four quality gates must be `"pass"` for the bead to be marked as complete. If any gate fails, the bead enters retry/context-wipe flow.
